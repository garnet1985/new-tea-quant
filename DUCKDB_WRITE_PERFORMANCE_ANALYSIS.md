# DuckDB vs ClickHouse 写入性能分析

## 🔍 问题诊断

### 当前性能问题

**实际测试结果**：
- **数据量**: 1700 万条记录
- **迁移时间**: 3 小时（10,800 秒）
- **写入速度**: **~1,574 行/秒** ⚠️
- **对比**: 多线程从 API 获取全量数据 < 1 小时（受 API 限流影响）

### 根本原因分析

**发现的问题**：迁移脚本使用了**逐行插入**，这是性能瓶颈！

```python
# ❌ 低效实现（当前代码）
for val in values:
    self.duckdb_conn.execute(insert_query, val)  # 每行一次 execute
```

**性能影响**：
- 每次 `execute()` 都有开销（SQL 解析、事务处理等）
- 1700 万行 = 1700 万次 `execute()` 调用
- 实际写入速度：~1,574 行/秒

**应该使用批量插入**：
```python
# ✅ 高效实现（修复后）
INSERT INTO table VALUES (row1), (row2), ..., (row10000)
```

## 📊 性能对比分析

### 1. DuckDB 写入性能（修复后）

- **小批量**（1000 行/批）: 10,000-50,000 行/秒
- **中批量**（10,000 行/批）: 50,000-200,000 行/秒
- **大批量**（100,000 行/批）: 100,000-500,000 行/秒

#### 优化后预期性能
- **1700 万数据迁移时间**: 
  - 保守估计（50,000 行/秒）: **5.7 分钟**
  - 理想情况（200,000 行/秒）: **1.4 分钟**
- **性能提升**: **32-128 倍**（相比当前逐行插入）

#### 限制因素
- **单连接写入**: DuckDB 是进程内数据库，单连接写入
- **内存限制**: 大批量插入需要足够内存
- **磁盘 I/O**: 写入性能受磁盘速度影响
