## 🎯 设计定位

### 核心目标

**专注于金融数据获取场景的轻量级、高性能数据提供者框架**

### 适用场景

✅ **最适合：**
- 中小规模数据团队（单机5000+只股票，完全够用）
- 金融数据获取（多数据源、精细限流、复杂依赖）
- 快速迭代开发
- 需要精细控制每个API的限流
- 初步开源项目

❌ **不适合（暂时）：**
- 超大规模分布式场景（10万+只股票）
- 需要Kubernetes编排
- 需要跨数据中心部署

### 性能评估

| 场景 | 单机能力 | 说明 |
|-----|---------|------|
| **股票K线** | 5000只 × 3周期 = 15000任务 | 多线程(4) + 限流(100/min) ≈ 45分钟 |
| **财务数据** | 5000只 | 多线程(4) + 限流(200/min) ≈ 12分钟 |
| **宏观数据** | 10个指标 | 单线程 ≈ 1分钟 |
| **复权因子** | 5000只 | 多线程(4) + 限流(1000/min) ≈ 5分钟 |

**总计：** 一次完整更新 ≈ 1小时（可接受）

## 💪 核心优势

### 1. 独创的API级别限流 ⭐⭐⭐⭐⭐

```python
# 其他框架：粗粒度限流（整个Spider、整个DAG）
# 我们：精细到每个API

rate_limit_registry.register_api('tushare.daily', 100)
rate_limit_registry.register_api('tushare.weekly', 50)
rate_limit_registry.register_api('tushare.monthly', 30)

# SmartConcurrentExecutor自动协调
# 限流差异大 → 自动选择串行，避免瓶颈
```

**价值：**
- 防止API封号（金融数据提供商限制严格）
- 最大化吞吐量（在限流约束下）
- 智能调度（自适应策略）

---

### 2. 优雅的声明式依赖

```python
# 简单声明
dependencies=[
    Dependency(
        provider='tushare',
        data_types=['stock_kline_daily']
    )
]

# 自动协调
await coordinator.coordinate_update('adj_factor', end_date)
# → 自动检查并更新 stock_kline_daily
```

**价值：**
- 零硬编码
- 自动拓扑排序
- 递归依赖处理

---

### 3. 轻量级、易于扩展

```python
# 新增Provider只需3步
1. class NewProvider(BaseProvider): pass
2. registry.mount('new_provider', NewProvider())
3. Done！

# 零修改核心代码
```

**价值：**
- 快速集成新数据源
- 适合开源项目（贡献者友好）
- 无需重启系统

## 🚫 已知限制

### 1. 单机限制（可接受）

**现状：**
- 单机多线程（4-8线程）
- 无分布式支持

**适用范围：**
- ✅ 5000只股票以内
- ✅ 日常增量更新
- ⚠️ 首次全量拉取可能较慢

**何时需要分布式：**
- 股票数 > 10000只
- 需要分钟级更新
- 需要跨地域部署

**届时方案：**
- 接入Celery（分布式任务队列）
- 或者集成到Airflow作为Operator

---

### 2. 缺少UI（计划补充）

**现状：**
- 只有命令行和日志
- 无可视化界面

**影响：**
- 调试稍显不便
- 需要一定技术能力

**计划：**
- Phase 6: 添加Web UI（FastAPI + React）
- 优先级：高，但不阻塞Phase 1-5

---

### 3. 缺少调度器（计划补充）

**现状：**
- 需要手动调用或外部cron

**影响：**
- 需要简单的外部调度

**计划：**
- Phase 7: 添加内置调度器
- 过渡方案：使用系统cron

## ⚠️ 已知权衡

### 1. 单机 vs 分布式

**选择：** 单机多线程

**原因：**
- 5000只股票完全够用
- 架构简单
- 适合开源项目初期

**未来：** 如有需要，可集成Celery

---

### 2. 内置调度 vs 外部调度

**选择：** 先外部调度（cron）

**原因：**
- 降低复杂度
- 专注核心功能
- cron够用

**未来：** Phase 7 添加内置调度器

---

### 3. 自研UI vs 无UI

**选择：** Phase 6 自研UI

**原因：**
- Phase 1-5 专注核心
- UI对开源很重要
- 不阻塞Phase 1-5

**技术栈：** FastAPI + React（已有ECharts）
