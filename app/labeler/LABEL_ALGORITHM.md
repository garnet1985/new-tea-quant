# 标签计算算法设计文档

## 概述

本文档描述了股票标签计算系统的核心算法设计，该算法采用循环推进策略，确保标签数据的连续性和准确性。

## 核心设计原则

1. **内存效率**：按需加载单股票K线数据，避免一次性加载所有股票数据
2. **数据准确性**：基于实际K线数据状态，避免因数据更新延迟导致的错误标签
3. **简洁有效**：使用hard code阈值，避免复杂配置和API调用
4. **容错性好**：失败就重试，不需要复杂的恢复机制

## 算法流程

### 1. 任务生成阶段

```
输入：股票列表、最新交易日(last_market_open_day)
输出：需要更新的股票任务列表

步骤：
1. 获取所有股票列表（过滤ST、科创板等）
2. 批量获取所有股票的最后标签更新日期
3. 判断哪些股票需要更新：
   - 从未更新过：需要更新
   - 距离上次更新 > 30天：需要更新
   - 距离上次更新 ≤ 30天：跳过
4. 为需要更新的股票生成job任务
```

### 2. 任务执行阶段

```
输入：单个股票的job任务
输出：更新后的标签数据

步骤：
1. 获取该股票的所有K线数据（一次性加载）
2. 执行循环推进算法
3. 保存计算结果到数据库
```

### 3. 循环推进算法（核心）

```
输入：股票ID、最后标签日期、所有K线数据、最新交易日
输出：计算出的标签数据

算法：
current_date = 最后标签日期
MIN_INTERVAL = 15  # hard code最小间隔阈值（天）

while True:
    # 计算目标日期
    target_date = current_date + 30天
    
    # 检查是否超过结束条件
    if target_date > last_market_open_day:
        break
    
    # 检查是否超过K线数据的最后日期
    if target_date > K线数据的最后日期:
        break
    
    # 查找实际交易日
    if target_date有对应的K线数据:
        actual_trading_day = target_date
    else:
        actual_trading_day = 找到target_date之前最近的K线日期
    
    # 阈值判断
    days_since_last_update = actual_trading_day - current_date
    if days_since_last_update < MIN_INTERVAL:
        # 距离太近，跳过本次计算
        current_date = actual_trading_day
        continue
    
    # 计算标签
    labels = 计算标签(actual_trading_day, 所有K线数据)
    if labels:
        保存标签到数据库(股票ID, actual_trading_day, labels)
    
    # 推进到下一个周期
    current_date = actual_trading_day
```

## 关键设计决策

### 1. 数据加载策略

- **Build Job阶段**：只加载股票列表 + 最近标签计算日期（轻量级）
- **Job执行阶段**：按需加载单只股票的所有K线数据
- **内存占用**：单线程 = 1只股票K线，多线程 = 线程数 × 1只股票K线

### 2. 跳过策略

- 如果股票停牌期间没有K线数据，选择跳过而不是计算标签
- 每次renew job都会包含这个股票，但推进时没有结果
- 避免因数据更新延迟导致的错误标签

### 3. 阈值判断

- 使用hard code的最小日期间隔（如15天）
- 避免过于频繁的标签计算
- 简单有效，避免复杂配置

### 4. 结束条件

- 主要基于`last_market_open_day`决定是否产生renew job
- 循环推进有两个结束条件：
  1. 超过`last_market_open_day`（市场最新交易日）
  2. 超过K线数据的最后日期（消耗完所有K线数据）
- 自然结束，无无限循环风险

## 异常处理策略

### 1. K线数据获取失败
- 在job一开始就执行，失败就直接退出job
- 下次执行job还会带上这个股票，因为数据没变化

### 2. 标签计算失败
- 认为不可能失败，纯代码计算逻辑
- 如果失败就改代码重新计算

### 3. 数据库写入失败
- 没关系，写入失败意味着数据不更新
- 不更新意味着下次还会变成job执行
- 这次出错下次还有机会

## 性能优化

1. **批量查询**：批量获取所有股票的最后更新日期
2. **按需加载**：单股票K线数据，避免内存问题
3. **避免API调用**：不判断停牌状态，避免性能损耗
4. **循环推进**：减少重复查询和计算

## 配置参数

- **更新间隔**：30天
- **最小阈值**：15天（hard code）
- **多线程阈值**：50个任务

## 数据流图

```
股票列表 → 判断需要更新 → 生成Job → 执行Job
    ↓
获取K线数据 → 循环推进算法 → 计算标签 → 保存到数据库
```

## 优势总结

1. **内存效率高**：按需加载，避免内存问题
2. **数据准确性好**：基于实际数据状态，避免错误标签
3. **逻辑清晰**：循环推进，确保不遗漏
4. **性能优化**：减少不必要的计算和存储
5. **简洁实用**：hard code参数，避免复杂配置
6. **容错性好**：失败就重试，自然恢复
