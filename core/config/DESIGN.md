# 配置系统设计

## 🎯 设计理念

### 核心原则

1. **配置与逻辑分离**
   - `core/config/` 只包含配置文件（JSON），不包含任何逻辑代码
   - 所有配置加载逻辑集中在 `ConfigManager` 中

2. **简单易用优先**
   - 普通用户只需配置必要信息（如数据库用户名和密码）
   - 高级配置（连接池、超时等）放在默认配置中，高级用户可在 userspace 中覆盖

3. **深度合并机制**
   - 用户配置会深度合并到默认配置，而不是完全替换
   - 支持部分覆盖，用户只需配置需要修改的部分

4. **环境变量支持**
   - 敏感信息（如密码）可通过环境变量覆盖
   - 环境变量优先级最高，适合生产环境

## 📋 设计背景

### 问题与挑战

1. **配置分散**：配置文件和加载逻辑混在一起，难以维护
2. **覆盖机制不完善**：用户配置完全替换默认配置，无法部分覆盖
3. **数据库配置复杂**：用户需要配置大量高级参数，使用门槛高
4. **配置与数据枚举混淆**：股票指数列表等数据枚举被当作配置管理

### 解决方案

1. **统一配置结构**：按功能拆分配置文件（data、system、market、worker、database）
2. **深度合并**：实现深度合并机制，支持部分覆盖
3. **简化数据库配置**：用户只需配置用户名和密码，其他使用默认值
4. **分离数据枚举**：将数据枚举清单移到 data_source 模块

## 🏗️ 架构设计

### 配置文件结构

```
core/config/                    # 配置文件目录（只包含 JSON）
├── data.json                   # 数据配置（默认开始日期、小数位数、股票过滤等）
├── market.json                 # 市场配置（未来扩展：T+0/T+1、做空支持等）
├── system.json                 # 系统配置
├── worker.json                 # Worker 配置（任务类型、核心数等）
└── database/                   # 数据库配置
    ├── common.json             # 公用配置（database_type、batch_write）
    ├── postgresql.json         # PostgreSQL 配置
    ├── mysql.json              # MySQL 配置
    └── sqlite.json              # SQLite 配置

userspace/config/               # 用户配置（可选，覆盖系统默认）
├── data.json                   # 用户数据配置
├── market.json                 # 用户市场配置
├── system.json                 # 用户系统配置
├── worker.json                 # 用户 Worker 配置
└── database/                   # 用户数据库配置
    ├── common.json             # 用户公用配置
    ├── postgresql.json         # 用户 PostgreSQL 配置（只需用户名和密码）
    ├── mysql.json              # 用户 MySQL 配置
    └── sqlite.json             # 用户 SQLite 配置
```

### 配置加载流程

1. **加载默认配置**：从 `core/config/` 加载系统默认配置
2. **加载用户配置**：从 `userspace/config/` 加载用户配置（如果存在）
3. **深度合并**：用户配置深度合并到默认配置
4. **环境变量覆盖**：环境变量覆盖配置值（最高优先级）

### 配置访问接口

所有配置通过 `ConfigManager` 统一访问：

- **完整配置接口**：`get_data_config()`, `get_database_config()` 等
- **便捷访问接口**：`get_default_start_date()`, `get_decimal_places()` 等

## 🔑 关键设计决策

### 1. 数据库配置简化

**设计**：用户只需配置用户名和密码，其他参数使用默认值

**原因**：
- 降低使用门槛，普通用户无需了解连接池等高级概念
- 高级用户仍可在 userspace 中覆盖高级配置

### 2. 深度合并机制

**设计**：用户配置深度合并到默认配置，而不是完全替换

**原因**：
- 支持部分覆盖，用户只需配置需要修改的部分
- 保持配置的完整性和一致性

### 3. 配置与逻辑分离

**设计**：`core/config/` 只包含配置文件，所有加载逻辑在 `ConfigManager` 中

**原因**：
- 清晰的职责划分，配置文件易于理解和修改
- 配置加载逻辑集中管理，便于维护和扩展

### 4. 市场配置抽象

**设计**：预留 `market.json` 配置，当前为空，为未来扩展预留

**原因**：
- 当前关注 A 股市场，未来可能需要支持其他市场
- 预留交易规则配置（T+0/T+1、做空支持等）

### 5. 数据枚举分离

**设计**：将股票指数列表等数据枚举移到 data_source 模块

**原因**：
- 数据枚举清单不是配置，而是数据定义
- 与配置系统分离，职责更清晰
