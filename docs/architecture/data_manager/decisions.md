本文档记录了 DataManager 模块的重要架构设计决策，包括问题背景、可选方案、决策理由和影响。

## 决策 1：Facade + Coordinator + Service 三层架构

**日期**：2025-12-19  
**状态**：已实施

### 痛点

在系统演进过程中，数据访问逻辑逐渐分散，出现了以下问题：
- 数据访问方法散落在多个模块，缺乏统一入口
- 业务逻辑与数据访问耦合，难以维护和测试
- 底层 Model 直接暴露，导致外部代码依赖实现细节
- 多次数据库查询效率低下，缺乏优化

### 可选方案

**方案 A：单层架构（所有逻辑在一个类中）**
- 优点：简单直接
- 缺点：职责不清，难以维护，代码臃肿

**方案 B：两层架构（Facade + Service）**
- 优点：职责分离，易于理解
- 缺点：跨服务协调逻辑无处放置

**方案 C：三层架构（Facade + Coordinator + Service）** ✅
- 优点：职责清晰，易于扩展，支持跨服务协调
- 缺点：层级稍多，学习成本略高

### 为什么选择当前方案

选择方案 C（三层架构）的理由：
1. **职责清晰**：Facade 负责生命周期管理，Coordinator 负责跨服务协调，Service 负责领域逻辑
2. **易于扩展**：添加新服务只需在对应层级添加，不影响其他层级
3. **支持复杂场景**：Coordinator 层可以处理需要跨多个服务的复杂请求（如 `prepare_data`）
4. **符合设计原则**：遵循单一职责原则和开闭原则

### 影响

- 代码结构清晰，易于理解和维护
- 新服务添加简单，只需在对应层级实现
- 跨服务协调逻辑集中在 Coordinator 层，便于管理

## 决策 2：严格职责分离，不提供便捷委托方法

**日期**：2025-12-19  
**状态**：已实施

### 痛点

在早期设计中，`StockService` 提供了便捷委托方法（如 `load_klines()`），导致：
- 职责不清：StockService 既负责基础信息，又负责 K线数据
- 方法冗余：同一个功能有多种访问方式，容易混淆
- 维护困难：修改子服务接口需要同步修改委托方法

### 可选方案

**方案 A：提供便捷委托方法**
- 优点：使用方便，代码简洁
- 缺点：职责不清，方法冗余，维护困难

**方案 B：严格职责分离，不提供委托方法** ✅
- 优点：职责清晰，方法唯一，易于维护
- 缺点：访问路径稍长（如 `data_mgr.stock.kline.load()`）

### 为什么选择当前方案

选择方案 B 的理由：
1. **明确性优先**：遵循 Python "Explicit is better than implicit" 原则
2. **职责清晰**：每个服务只负责自己的领域，不越界
3. **易于维护**：修改子服务接口不需要修改父服务
4. **降低学习成本**：用户只需要记住一种访问方式

### 影响

- 访问路径稍长，但意图更明确
- 代码维护更容易，职责边界清晰
- 用户需要明确指定服务路径（这是期望的行为）

## 决策 3：底层 Model 完全私有化

**日期**：2025-12-19  
**状态**：已实施

### 痛点

在早期设计中，底层 Model 类直接暴露给外部代码，导致：
- 实现细节泄露：外部代码依赖 Model 的内部实现
- 难以重构：修改 Model 会影响所有使用它的代码
- 行为不一致：不同模块可能以不同方式访问数据

### 可选方案

**方案 A：Model 类直接暴露**
- 优点：访问直接，性能好
- 缺点：实现细节泄露，难以重构，行为不一致

**方案 B：Model 类完全私有化，只能通过 DataService 访问** ✅
- 优点：实现细节隐藏，易于重构，行为一致
- 缺点：需要额外的 Service 层，性能略低（可忽略）

### 为什么选择当前方案

选择方案 B 的理由：
1. **解耦**：外部代码不依赖底层 Model 实现，只依赖稳定的 DataService API
2. **灵活性**：可以自由重构 Model 层，而不影响外部代码
3. **一致性**：所有数据访问都通过统一的 DataService 接口，保证行为一致
4. **可维护性**：清晰的边界使得代码更容易理解和维护

### 影响

- 外部代码无法直接访问 Model，必须通过 DataService
- Model 层可以自由重构，不影响外部代码
- 所有数据访问行为一致，便于维护

## 决策 4：使用 SQL JOIN 优化查询

**日期**：2025-12-19  
**状态**：已实施

### 痛点

在早期实现中，QFQ（前复权）查询需要多次数据库访问：
1. 查询 K线数据
2. 查询复权因子
3. 在应用层合并数据

这导致：
- 性能问题：多次数据库往返，延迟高
- 网络开销：多次网络请求
- 代码复杂：需要在应用层处理数据合并

### 可选方案

**方案 A：多次查询，应用层合并**
- 优点：实现简单
- 缺点：性能差，网络开销大，代码复杂

**方案 B：使用 SQL JOIN 一次查询** ✅
- 优点：性能好，网络开销小，代码简单
- 缺点：SQL 查询稍复杂

### 为什么选择当前方案

选择方案 B 的理由：
1. **性能优化**：一次查询获取所有需要的数据，减少数据库往返
2. **网络效率**：减少网络请求次数，降低延迟
3. **代码简化**：数据库层面完成数据关联，应用层代码更简洁
4. **数据库优势**：利用数据库的优化能力（索引、查询优化器等）

### 影响

- 查询性能显著提升（从多次查询变为一次查询）
- SQL 查询稍复杂，但数据库可以优化
- 代码更简洁，数据关联逻辑在数据库层完成

## 决策 5：自动初始化设计

**日期**：2025-12-19  
**状态**：已实施

### 痛点

在早期设计中，用户需要手动调用 `initialize()` 方法，导致：
- 使用不便：容易忘记初始化
- 错误频发：未初始化就使用会导致错误
- 代码冗余：每次使用都需要初始化

### 可选方案

**方案 A：手动初始化**
- 优点：控制灵活
- 缺点：使用不便，容易出错

**方案 B：自动初始化（在 __init__ 中调用）** ✅
- 优点：使用方便，不易出错
- 缺点：初始化时机不可控（可通过显式调用解决）

### 为什么选择当前方案

选择方案 B 的理由：
1. **使用方便**：创建实例后即可使用，无需手动初始化
2. **减少错误**：避免忘记初始化导致的错误
3. **幂等性保证**：`initialize()` 方法是幂等的，多次调用只执行一次
4. **灵活性保留**：保留显式 `initialize()` 方法，支持特殊场景

### 影响

- 使用更方便，创建后即可使用
- 减少初始化相关的错误
- 特殊场景仍可通过显式调用控制初始化时机

## 决策 6：进程级单例模式

**日期**：2025-12-19  
**状态**：已实施

### 痛点

在多进程环境下，如果使用全局单例，会导致：
- 进程间共享问题：多进程环境下进程间内存不共享
- 初始化问题：子进程需要重新初始化
- 资源管理问题：连接池等资源需要在每个进程独立管理

### 可选方案

**方案 A：全局单例（跨进程共享）**
- 优点：只有一个实例
- 缺点：多进程环境下不可行，资源管理复杂

**方案 B：进程级单例（每个进程独立实例）** ✅
- 优点：多进程友好，资源管理简单
- 缺点：每个进程有独立实例（这是期望的行为）

### 为什么选择当前方案

选择方案 B 的理由：
1. **多进程友好**：每个进程有独立的实例，避免进程间共享问题
2. **资源管理简单**：每个进程独立管理自己的数据库连接池
3. **符合 Python 多进程模型**：进程间内存不共享，单例也应该是进程级的
4. **线程安全**：使用线程锁保证单例创建的线程安全

### 影响

- 多进程环境下每个进程有独立的 DataManager 实例
- 资源（如连接池）在每个进程独立管理
- 线程安全，支持多线程环境

## 决策 7：Schema 驱动的表初始化

**日期**：2025-12-19  
**状态**：已实施

### 痛点

在早期设计中，表结构定义分散在代码中，导致：
- 表结构定义不清晰：需要在代码中查找 CREATE TABLE 语句
- 难以维护：修改表结构需要修改代码
- 多数据库支持困难：不同数据库的 SQL 语法不同
- 缺乏文档：表结构没有统一的文档格式

### 可选方案

**方案 A：代码中定义表结构（CREATE TABLE 语句）**
- 优点：实现简单
- 缺点：不清晰，难以维护，多数据库支持困难

**方案 B：使用 ORM（如 SQLAlchemy）**
- 优点：功能强大，支持迁移
- 缺点：学习成本高，性能开销，过度设计

**方案 C：使用 JSON Schema 声明式定义** ✅
- 优点：清晰、易维护、易扩展、多数据库支持
- 缺点：需要实现 Schema 到 SQL 的转换逻辑

### 为什么选择当前方案

选择方案 C 的理由：
1. **声明式设计**：Schema 是表结构的声明，不包含实现细节
2. **易于维护**：修改表结构只需修改 JSON 文件
3. **多数据库支持**：SchemaManager 可以根据数据库类型生成对应的 SQL
4. **文档化**：Schema 文件本身就是文档，清晰易懂
5. **自动化**：系统启动时自动读取 Schema 并创建表

### 影响

- 表结构定义清晰，易于理解和维护
- 支持多数据库（PostgreSQL、MySQL、SQLite）
- 系统启动时自动创建表，无需手动执行 SQL

## 决策 8：每个 Schema 对应一个 Model

**日期**：2025-12-19  
**状态**：已实施

### 痛点

在早期设计中，使用通用的 Model 类操作所有表，导致：
- 缺乏类型安全：无法区分不同表的操作
- 业务方法无处放置：常用的业务查询方法无法封装
- 代码重复：每个使用的地方都需要写相同的查询逻辑

### 可选方案

**方案 A：使用通用 Model（DbBaseModel 直接实例化）**
- 优点：简单，不需要为每个表创建类
- 缺点：缺乏类型安全，业务方法无处放置，代码重复

**方案 B：每个表创建独立的 Model 类** ✅
- 优点：类型安全，可以封装业务方法，代码复用
- 缺点：需要为每个表创建类（但这是值得的）

### 为什么选择当前方案

选择方案 B 的理由：
1. **职责单一**：每个 Model 只负责一个表的操作，职责清晰
2. **业务方法封装**：可以在 Model 中封装常用的业务查询方法
3. **类型安全**：每个 Model 对应一个表，类型明确，IDE 支持更好
4. **代码复用**：业务查询方法可以在 Model 中复用，避免重复代码

### 影响

- 需要为每个表创建 Model 类（但代码量不大）
- 业务查询方法可以封装在 Model 中，提高代码复用性
- 类型更安全，IDE 支持更好

## 决策 9：Model 继承 DbBaseModel

**日期**：2025-12-19  
**状态**：已实施

### 痛点

如果 Model 不继承基类，每个 Model 都需要实现：
- 基础的 CRUD 操作（load、save、delete、update）
- 批量操作
- 分页查询
- 参数化查询（防 SQL 注入）
- Schema 加载逻辑
- 数据库适配逻辑

这会导致：
- 代码重复：每个 Model 都需要实现相同的逻辑
- 维护困难：修改基础功能需要修改所有 Model
- 容易出错：每个实现可能有细微差异

### 可选方案

**方案 A：每个 Model 独立实现所有功能**
- 优点：灵活性高
- 缺点：代码重复，维护困难，容易出错

**方案 B：Model 继承 DbBaseModel，复用通用功能** ✅
- 优点：代码复用，维护简单，行为一致
- 缺点：需要理解继承关系（但这是标准做法）

### 为什么选择当前方案

选择方案 B 的理由：
1. **代码复用**：通用的 CRUD 操作在基类中实现，所有 Model 复用
2. **维护简单**：修改基础功能只需修改基类，所有 Model 自动受益
3. **行为一致**：所有 Model 的行为一致，减少 bug
4. **功能丰富**：基类提供了丰富的功能（批量操作、分页查询、时序数据优化等）

### 影响

- Model 实现更简单，只需关注业务特定的查询方法
- 基础功能统一维护，行为一致
- 新功能可以在基类中添加，所有 Model 自动获得

## 决策 10：引入 Service 层

**日期**：2025-12-19  
**状态**：已实施

### 痛点

如果只有 Model 层，会出现以下问题：
- **跨表查询困难**：Model 只负责单表操作，跨表查询需要在业务代码中实现
- **业务逻辑分散**：业务逻辑散落在各个使用 Model 的地方
- **数据转换重复**：每个使用的地方都需要做相同的数据转换
- **实现细节暴露**：业务代码需要知道底层有哪些表，如何组合

### 可选方案

**方案 A：只有 Model 层，业务代码直接使用 Model**
- 优点：简单直接
- 缺点：跨表查询困难，业务逻辑分散，代码重复

**方案 B：引入 Service 层，封装业务逻辑** ✅
- 优点：业务逻辑集中，跨表查询简单，数据转换统一
- 缺点：增加一层抽象（但这是值得的）

### 为什么选择当前方案

选择方案 B 的理由：
1. **业务逻辑集中**：Service 封装领域特定的业务逻辑，业务代码更简洁
2. **跨表查询简单**：Service 可以组合多个 Model 完成复杂查询
3. **数据转换统一**：Service 统一处理数据格式转换，避免重复代码
4. **封装实现细节**：Service 隐藏底层表结构，外部代码不依赖实现细节

### 影响

- 增加了一层抽象，但业务代码更简洁
- 业务逻辑集中在 Service 层，易于维护
- 可以自由重构底层表结构，不影响外部代码

## 决策 11：淘汰 prepare_data 方法

**日期**：2026-01-17  
**状态**：已淘汰

### 痛点

早期设计中，`prepare_data` 方法用于解析 Tag 或 Strategy 的 settings 配置，根据配置自动加载所需数据。但在实际使用中发现：

1. **数据加载逻辑不同**：Tag 和 Strategy 模块的数据加载需求差异很大
   - **Tag 模块**：需要支持 chunk 模式（滑动窗口）、增量更新、lookback 数据等复杂场景
   - **Strategy 模块**：需要全量加载、游标机制逐日过滤、支持多周期数据等不同场景
   
2. **无法统一抽象**：两种场景的数据加载逻辑差异太大，无法用一个通用的 `prepare_data` 方法满足

3. **实际未使用**：Tag 和 Strategy 模块都实现了各自的数据管理器（`TagWorkerDataManager`、`StrategyWorkerDataManager`），不依赖 `prepare_data`

### 可选方案

**方案 A：保留 prepare_data，继续完善**
- 优点：统一的 API
- 缺点：无法满足不同场景的复杂需求，代码会变得臃肿

**方案 B：为每个模块实现独立的数据管理器** ✅
- 优点：可以针对不同场景优化，代码清晰
- 缺点：代码重复（但可以通过基类减少重复）

### 为什么选择当前方案

选择方案 B 的理由：
1. **场景差异大**：Tag 和 Strategy 的数据加载需求差异太大，无法统一抽象
2. **性能优化**：独立的数据管理器可以针对各自场景进行性能优化（如 Tag 的 chunk 模式、Strategy 的游标机制）
3. **职责清晰**：每个模块的数据管理器专注于自己的场景，职责清晰
4. **易于维护**：独立实现，互不影响，易于维护和扩展

### 影响

- `prepare_data` 方法被标记为淘汰，不再推荐使用
- Tag 模块使用 `TagWorkerDataManager` 处理数据加载
- Strategy 模块使用 `StrategyWorkerDataManager` 处理数据加载
- 两个数据管理器都继承或使用 DataManager 的底层能力（Model 层），但数据加载逻辑独立实现
